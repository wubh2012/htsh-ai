# 合同智能审核系统 - MVP设计文档

## 1. 项目概述

### 1.1 项目目标
构建一个合同智能审核MVP，用户上传Word或PDF合同文档后，系统调用AI模型根据预设规则进行审核，返回审核结论和修改建议，最终由人工复核确认。

### 1.2 MVP范围
- **文档格式**：Word(.docx)、文字型PDF
- **规则类型**：条款缺失检查(CHECK_MISSING)、风险关键词检查(RISK_KEYWORD)
- **AI调用**：支持配置不同AI API（通义千问、智谱等）
- **审核流程**：上传 → AI审核 → 结果展示 → 人工复核

### 1.3 后期扩展预留
- 更多规则类型（数值范围、语义风险）
- 更多文档格式支持
- 批量审核
- 规则模板系统

### 2.1 技术栈

## 3. 核心模块设计

### 3.1 文档解析模块 (services/parser.py)

**功能**：解析Word和PDF文档，提取纯文本

**支持格式**：
- Word: .docx (python-docx)
- PDF: 文字型PDF (pymupdf)

**接口设计**：
```python
async def parse_document(file_path: str, file_type: str) -> str:
    """
    解析文档并返回纯文本
    :param file_path: 文件路径
    :param file_type: "docx" 或 "pdf"
    :return: 提取的纯文本
    """
```

**文本清洗**：
- 去除多余空白字符
- 还原换行格式
- 处理特殊编码字符

### 3.2 AI调用模块 (services/ai_client.py)

**功能**：封装AI API调用，支持配置不同provider

**支持的AI Provider**：
- 智谱 (zhipuai)
- 通义千问 (dashscope)
- 百度文心一言 (wenxin)
- OpenAI GPT (openai)

**接口设计**：
```python
class AIClient:
    async def chat(self, messages: list, provider: str = "zhipuai") -> str:
        """
        调用AI模型
        :param messages: 消息列表
        :param provider: AI提供商
        :return: AI返回的文本
        """
```

**配置方式**：config.py中通过字典配置API密钥和endpoint

### 3.3 审核逻辑模块 (services/auditor.py)

**功能**：将规则和合同文本组合成prompt，调用AI并解析结果

**审核流程**：
1. 加载启用的规则列表
2. 将规则格式化为prompt
3. 构建审核prompt（包含合同文本、规则、输出格式要求）
4. 调用AI模型
5. 解析AI返回的JSON结果
6. 存储审核结果

**Prompt设计**：
```
你是一个专业的合同审核律师。请根据以下规则审核合同内容。

【审核规则】
{规则列表，每条规则包含：名称、类型、检查内容、风险等级、修改建议}

【合同内容】
{合同文本}

【输出要求】
请以JSON格式返回审核结果，包含：conclusion(结论)、riskPoints(风险点列表)、summary(总体评价)
```

### 3.4 规则配置模块 (routers/rule.py)

**功能**：规则的增删改查

**API接口**：
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/rules | 获取所有规则 |
| GET | /api/rules/{id} | 获取单个规则 |
| POST | /api/rules | 创建规则 |
| PUT | /api/rules/{id} | 更新规则 |
| DELETE | /api/rules/{id} | 删除规则 |
| PATCH | /api/rules/{id}/toggle | 启用/禁用规则 |

### 3.5 审核流程模块 (routers/audit.py)

**功能**：合同上传、审核发起、结果查询、复核操作

**API接口**：
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/upload | 上传合同文档 |
| POST | /api/audit/{result_id} | 发起AI审核 |
| GET | /api/audit/{result_id} | 获取审核结果 |
| POST | /api/audit/{result_id}/approve | 人工通过 |
| POST | /api/audit/{result_id}/reject | 人工驳回 |
| GET | /api/audit/list | 审核列表 |

## 4. 数据库设计

### 4.1 规则表 (audit_rule)

```sql
CREATE TABLE audit_rule (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_name VARCHAR(100) NOT NULL,      -- 规则名称
    rule_type VARCHAR(50) NOT NULL,       -- CHECK_MISSING / RISK_KEYWORD
    check_content VARCHAR(500) NOT NULL,  -- 检查的条款名称或关键词
    risk_level INTEGER DEFAULT 3,         -- 风险等级 1-5
    suggestion TEXT,                       -- 修改建议
    enabled BOOLEAN DEFAULT TRUE,          -- 是否启用
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 4.2 审核结果表 (audit_result)

```sql
CREATE TABLE audit_result (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    contract_name VARCHAR(200) NOT NULL, -- 合同文件名
    file_path VARCHAR(500) NOT NULL,      -- 文件存储路径
    file_type VARCHAR(10) NOT NULL,        -- docx / pdf
    content_text TEXT,                     -- 提取的合同文本
    ai_conclusion VARCHAR(20),             -- AI结论：PASS / FAIL / REVIEW
    ai_result JSON,                        -- AI返回的详细结果
    audit_status VARCHAR(20) DEFAULT 'PENDING',  -- PENDING / APPROVED / REJECTED
    auditor_comment TEXT,                   -- 复核意见
    auditor_id INTEGER,                    -- 复核人ID
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    audit_time DATETIME                    -- 复核时间
);
```

### 4.3 AI配置表 (ai_config)

```sql
CREATE TABLE ai_config (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    provider VARCHAR(50) NOT NULL,        -- zhipuai / dashscope / wenxin / openai
    api_key VARCHAR(200) NOT NULL,         -- API密钥
    endpoint VARCHAR(500),                -- API endpoint（可选）
    model VARCHAR(100),                    -- 模型名称
    enabled BOOLEAN DEFAULT FALSE,         -- 是否启用
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 5. API详细设计

### 5.1 规则配置API

#### 创建规则
```
POST /api/rules
Request Body:
{
    "rule_name": "保密条款检查",
    "rule_type": "CHECK_MISSING",
    "check_content": "保密条款",
    "risk_level": 3,
    "suggestion": "建议在合同中添加保密条款，明确保密范围和保密期限"
}

Response:
{
    "id": 1,
    "rule_name": "保密条款检查",
    ...
}
```

#### 获取规则列表
```
GET /api/rules

Response:
{
    "rules": [
        {"id": 1, "rule_name": "保密条款检查", ...},
        {"id": 2, "rule_name": "违约金检查", ...}
    ]
}
```

### 5.2 审核API

#### 上传合同
```
POST /api/upload
Content-Type: multipart/form-data

file: [文件]

Response:
{
    "result_id": 1,
    "contract_name": "合同示例.docx",
    "message": "上传成功"
}
```

#### 发起审核
```
POST /api/audit/1

Response:
{
    "result_id": 1,
    "ai_conclusion": "REVIEW",
    "ai_result": {
        "conclusion": "REVIEW",
        "riskPoints": [
            {
                "ruleId": 1,
                "ruleName": "保密条款检查",
                "description": "缺失保密条款",
                "suggestion": "建议添加..."
            }
        ],
        "summary": "合同存在X个风险点，建议修改后提交人工复核"
    }
}
```

#### 人工复核
```
POST /api/audit/1/approve
{
    "auditor_comment": "已确认，风险可控"
}

POST /api/audit/1/reject
{
    "auditor_comment": "驳回，需要修改第X条"
}
```

## 7. 关键技术难点与解决方案

### 7.1 PDF解析质量
- **问题**：不同来源的PDF格式差异大
- **解决**：MVP阶段只支持文字型PDF，提供解析失败提示

### 7.2 AI输出格式不稳定
- **问题**：LLM输出JSON可能格式混乱
- **解决**：
  - 要求AI用markdown代码块包裹JSON
  - 后端做容错解析，提取JSON部分
  - 提供解析失败fallback

### 7.3 长文本处理
- **问题**：合同可能很长，超过AI模型的context limit
- **解决**：MVP阶段限制合同大小，按固定字数分段审核（后期优化）

### 7.4 AI API配置灵活性
- **问题**：需要支持多个AI provider
- **解决**：通过数据库配置AI provider，支持切换
